# 커밋 작업 단위 운영 기준

## 목적

- 기능 개발이 일정 수준에 도달했을 때 바로 커밋/푸시해 변경 이력을 짧고 명확하게 유지한다.
- "한 번에 너무 큰 커밋"을 피하고, 되돌리기/리뷰가 쉬운 단위로 분리한다.

## 기본 원칙

1. 한 커밋은 한 가지 의도만 담는다.
   - 예: "시나리오 계측 이벤트 추가"
   - 비예: "계측 + 문서 + UI 전면개편" 혼합

2. 사용자 가치 기준으로 끊는다.
   - 사용자가 체감 가능한 변화 1개가 완성되면 커밋한다.

3. 검증 통과 후 커밋한다.
   - 최소: `npm run lint`
   - 기능/구조 변경: `npm run lint && npm run build`

4. 문서 동기화는 같은 라운드에서 별도 커밋으로 분리 가능하다.
   - 코드 커밋
   - 문서 커밋

## 이 저장소 권장 커밋 단위

- `feat`: 기능 추가/행동 흐름 개선
- `fix`: 버그 수정(런타임 오류, 상태 복원 오류, key 중복 등)
- `docs`: 계획/가이드/요구사항/리뷰 문서 갱신
- `refactor`: 동작 변경 없이 구조 개선

## 분리 기준 체크리스트

- 디렉터리가 크게 다르면 분리 (`src/**` vs `review_docs/**`)
- 롤백 대상이 다르면 분리
- 리뷰어가 1분 안에 의도를 설명하기 어렵다면 분리
- 커밋 메시지에 "그리고"가 붙으면 분리

## 커밋 타이밍 규칙

1. 구현 단위 완료
   - 예: 이벤트 스키마 + 소비 코드 연결 완료

2. 로컬 검증 완료
   - lint/build 성공

3. 문서 반영 여부 확인
   - 변경이 UX/API/운영 정책에 영향이 있으면 `review_docs`/`README` 갱신

4. 즉시 커밋/푸시
   - main 운영 시 작은 단위로 자주 푸시

## 이번 라운드 적용 예시

- 코드 커밋
  - 시나리오/워치리스트 계측 이벤트 확장
  - 브리핑 필터 복원 재도입
  - watchlist "비교/상황추천" CTA 추가

- 문서 커밋
  - 복원/볼륨제어 스트레스 체크리스트
  - 신뢰도/신선도 문구 가이드
  - 본 커밋 작업 단위 운영 기준 문서
